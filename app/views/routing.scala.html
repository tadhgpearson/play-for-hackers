Routes and Routing

Lets see how this page is displayed

The main entry point of your application is the conf/routes file. This file defines all of the application's accessible URLs.

GET /    controllers.Slides.index()

That simply tells Play that when the web server receives a GET request for the / path, it must call the controllers.Slides.index() method, you can see that method on the left

Here, the action returns a 200 OK response with an HTML response body. The HTML content is provided by a Scala template compiled to standard Java methods, here as views.html.index.render(String title).

  • Play will auto-detect the output content-type based on the object you provide to the result method,
    eg. String --> text/plain, JSONNode object --> text/json
  • HTML input parameters can be URL parameters or part of the URL itself, configurable in the routes file.
    eg. GET /tasks/:id controllers.Application.tasks(id: Long)
  • API for storing/reading cookies
  • You can have as many routes files as you want, add more in your application.conf
  • You can do reverse routing, implementers of routes can call back to the routes using their own input parameters
  • The routing syntax supports regular expressions, write your own instantly-unmaintainable program in the routes file!

Restful APIs

Obviously this isn't just useful for webpages, you can also build your own restful server APIs easily like this, and then call them from your rich UI framework, delivering JSON objects back to the your client.

  • GET, POST, PUT, DELETE as you need
  • Read POST parameters using the DynamicForm object, which happens to look exactly like a DataMap.. or accept a JSON objects over Jackson
